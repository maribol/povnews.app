How to write `whyItMatters`, `summary`, and `quotableSnippet` for digest items. The user's voice comes from their **User POV** JSON (`about`, pillars, audiences, and any learned preferences).

## Voice DNA for digest copy

1. **Open `whyItMatters` with relevance, not recap.** Not "Company X announced Y." A specific reason the user should care given their POV.

2. **Name the problem class, not the competitor.** The enemy is behavior or system failure â€” tool sprawl, broken handoffs, fragile testing â€” not "[Competitor] is bad."

3. **Be specific.** Use concrete nouns the user's audience recognizes â€” not abstract filler like "workflow optimization" or "digital transformation."

4. **Stay honest.** No hype, fake urgency, or invented stats. Only quote what the source actually says.

5. **Keep it short.** `whyItMatters` is one sentence. `summary` is 2â€“3 factual sentences. `quotableSnippet` is a short verbatim quote when available.

## Tone characteristics

| Characteristic | In practice |
|---------------|------------|
| **Direct** | No throat-clearing. First clause earns attention. |
| **Specific** | Tie the news to a decision or pain the user actually has. |
| **Practitioner-level** | Assume the reader knows their domain. No dumbing down. |
| **Concise** | One sentence for `whyItMatters`. Short summaries. |

Adjust examples and vocabulary to match the user's `about` text and `audiences[]`.

## Banned words and phrases

Avoid these in `whyItMatters` and `summary`:

| Banned | Use instead |
|--------|-------------|
| Leverage | Use |
| Empower | Give you, let you |
| Streamline | Speed up, simplify, remove steps |
| Robust | Describe the specific capability |
| Cutting-edge / Revolutionary / Game-changing | Describe what actually changed |
| Best-in-class / Powerful / Seamless | Show evidence instead |
| Unlock | Get access to, start using |
| All-in-one | Full-stack, complete platform |
| Disrupt / Next-level / Supercharge | Solve, fix, improve â€” with specifics |
| No-brainer | Describe why it matters |
| In today's fast-paced world | Delete |

## Audience-aware language

Use vocabulary that matches the user's `audiences[]` from User POV â€” mirror their domain jargon from `about` and pillar descriptions rather than defaulting to generic business speak.

When unsure, pull phrasing directly from User POV `about` and the matched pillar's `description`.

## Learned preferences

If the User POV includes **Pro examples** or **Anti-patterns** from feedback, treat them as hard guidance:

- Prefer angles similar to pro examples
- Avoid patterns listed as anti-patterns
- Do not repeat URLs or theses the user has already dismissed
